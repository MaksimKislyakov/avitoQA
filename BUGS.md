### <a id="bug-1"></a>BUG-1: Нарушение контракта ответа POST /api/1/item
- **Связанный тест-кейс:** ([ТК-1](TESTCASES.md#tk-1)) (Успешное создание объявления со всеми валидными полями)
- **Приоритет:** High
- **Эндпоинт:** `POST /api/1/item`
- **Описание:** Согласно Postman-коллекции, сервер должен возвращать JSON-объект с полями `id`, `sellerId`, `name`, `price`, `statistics`, `createdAt`. Фактически сервер возвращает объект `{"status": "Сообщение с ID"}`, из которого невозможно программно получить остальные поля созданного объекта без парсинга строки.
- **Ожидаемый результат:** `{ "id": "...", "sellerId": 123322, "name": "...", ... }`
- **Фактический результат:** `{ "status": "Сохранили объявление - ..." }`

### <a id="bug-9"></a>BUG-9: Некорректный формат даты в поле createdAt
- **Связанный тест-кейс:** [ТК-12](TESTCASES.md#tk-12) (Успешное получение существующего объявления)
- **Приоритет:** Medium
- **Эндпоинт:** `GET /api/1/item/:id`, `GET /api/1/:sellerID/item`
- **Описание:** Поле `createdAt` возвращается в формате `"YYYY-MM-DD HH:mm:ss.ssssss +0300 +0300"`. Это нарушает стандарт ISO 8601 (отсутствует разделитель `'T'`, часовой пояс продублирован). Это приведет к ошибкам парсинга на клиенте.
- **Ожидаемый результат:** `"2026-04-20T13:38:19.617Z"` или `"2026-04-20T13:38:19.617+03:00"`
- **Фактический результат:** `"2026-04-20 13:38:19.617609 +0300 +0300"`

### <a id="bug-11"></a>BUG-11: Сохранение статистики после удаления объявления
- **Связанный тест-кейс:** [ТК-24](TESTCASES.md#tk-24) (Проверка каскадного удаления статистики)
- **Приоритет:** High
- **Эндпоинт:** `DELETE /api/2/item/:id`, `GET /api/1/statistic/:id`
- **Описание:** при удалении объявления через API v2, само объявление корректно удаляется (`GET /api/1/item/:id` возвращает `404`), но статистика по этому ID остается доступной и возвращается со статусом `200 OK`. Нарушена целостность данных: удаленный объект не должен иметь доступную статистику
- **Ожидаемый результат:** статус `404 Not Found` при запросе статистики удаленного объявления
- **Фактический результат:** статус `200 OK`, статистика возвращается

### <a id="bug-13"></a>BUG-13: Успешное создание объявления без заголовка Content-Type
- **Связанный тест-кейс:** [ТК-31](TESTCASES.md#tk-31) (Отправка POST без заголовка Content-Type: application/json)
- **Приоритет:** Medium
- **Эндпоинт:** `POST /api/1/item`
- **Описание:** сервер успешно обрабатывает POST-запрос с телом в формате JSON, даже если заголовок `Content-Type` отсутствует или установлен в `text/plain`. Ожидается строгая валидация заголовков.
- **Ожидаемый результат:** статус `400 Bad Request` или `415 Unsupported Media Type`
- **Фактический результат:** статус `200 OK`, объявление создано
