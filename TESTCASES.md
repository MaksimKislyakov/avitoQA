# 1. Создание объявления: POST /api/1/item

## Позитивные сценарии

### <a id="tk-1"></a>Тест-кейс 1: Успешное создание объявления со всеми валидными полями

# 5. E2E (Сквозные сценарии проверки бизнес-логики)

### Тест-кейс 23: Полный жизненный цикл объявления
- **Предусловия:** нет
- **Шаги:** 
  1. Создать объявление (`POST /api/1/item`). 
  2. Получить его по ID (`GET /api/1/item/:id`). 
  3. Получить статистику (`GET /api/1/statistic/:id`). 
  4. Проверить, что оно есть в списке продавца (`GET /api/1/:sellerID/item`)
- **Ожидаемый результат:** все 4 запроса возвращают 200 OK, данные согласованы между ответами
- **Статус:** passed

### <a id="tk-24"></a>Тест-кейс 24: Проверка каскадного удаления статистики (E2E Cross-Version)
- **Предусловия:** нет
- **Шаги:** 
  1. Создать объявление (`POST /api/1/item`). 
  2. Удалить его через API v2 (`DELETE /api/2/item/:id`). 
  3. Убедиться, что объявление недоступно (`GET /api/1/item/:id -> 404`). 
  4. Проверить доступность статистики удаленного объявления (`GET /api/1/statistic/:id`)
- **Ожидаемый результат:** при запросе статистики удаленного объявления должен возвращаться статус 404 Not Found или пустой ответ, так как данные объявлений и статистики должны удаляться синхронно
- **Статус:** failed (см. [BUG-11](BUGS.md#bug-11))

### Тест-кейс 25: Проверка консистентности списка продавца после удаления объявления
- **Предусловия:** нет
- **Шаги:** 
  1. Создать два объявления для одного продавца (`POST /api/1/item`). 
  2. Получить список объявлений продавца (`GET /api/1/{sellerID}/item`) и убедиться, что их количество >= 2. 
  3. Удалить первое объявление через API v2 (`DELETE /api/2/item/:id`). 
  4. Снова запросить список объявлений продавца
- **Ожидаемый результат:** статус 200 OK. В обновленном списке отсутствует удаленное объявление, но сохраняется второе. Список синхронизирован с операцией удаления, данные не дублируются и не теряются
- **Статус:** passed

### Тест-кейс 26: Согласованность статистики между версиями API v1 и v2
- **Предусловия:** нет
- **Шаги:** 
  1. Создать объявление (`POST /api/1/item`). 
  2. Запросить статистику через API v1 (`GET /api/1/statistic/:id`). 
  3. Запросить статистику через API v2 (`GET /api/2/statistic/:id`). 
  4. Сравнить значения метрик (`likes`, `viewCount`, `contacts`) в обоих ответах
- **Ожидаемый результат:** статус 200 OK для обоих запросов. Значения полей метрик полностью идентичны в ответах v1 и v2, что подтверждает использование единого источника данных без рассинхронизации
- **Статус:** passed

### Тест-кейс 27: Изоляция данных между разными продавцами
- **Предусловия:** нет
- **Шаги:** 
  1. Создать объявление для Продавца А (`POST /api/1/item` с уникальным `sellerID_A`). 
  2. Создать объявление для Продавца Б (`POST /api/1/item` с уникальным `sellerID_B`). 
  3. Запросить список объявлений для Продавца А. 
  4. Запросить список объявлений для Продавца Б
- **Ожидаемый результат:** статус 200 OK для обоих запросов. В списке Продавца А присутствует только его объявление и полностью отсутствуют товары Продавца Б, и наоборот. Фильтрация по `sellerID` работает строго и не допускает перекрестного доступа к данным
- **Статус:** passed
